# 技能编排器 (Skill Orchestrator)

智能协调其他技能的调用时机和顺序，确保开发流程规范且高效。

## 核心理念

**在合适的时机，用合适的方式，调用合适的技能。**

不是强制执行，而是智能引导；不是剥夺判断，而是增强决策。

## 平台兼容性

### Claude Code
使用 `Skill` 工具调用技能：
```
调用方式：使用 Skill 工具，传入技能名称
示例：Skill tool with skill="skill-name"
```

### OpenCode
根据平台配置加载技能：
- 检查 `~/.opencode/skills/` 目录
- 或使用平台指定的技能加载机制

## 工作流程

```
┌─────────────────────────────────────────────────────────────┐
│                    用户消息到达                              │
└─────────────────────────┬───────────────────────────────────┘                          ▼
┌─────────────────────────────────────────────────────────────┐
│              第一步：快速判断（< 1秒）                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 这是简单对话/问候/信息查询吗？                         │   │
│  │ → 是：直接响应，跳过技能检查                           │   │
│  │ → 否：进入第二步                                      │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────┬───────────────────────────────────┘                          ▼
┌─────────────────────────────────────────────────────────────┐
│              第二步：技能匹配分析                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 分析任务类型：                                        │   │
│  │ • 新功能开发 → brainstorming → writing-plans         │   │
│  │ • Bug 修复 → debugging                               │   │
│  │ • 代码审查 → code-review                             │   │
│  │ • 文档编写 → documentation                           │   │
│  │ • 测试相关 → test-driven-development                 │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────┬───────────────────────────────────┘                          ▼
┌─────────────────────────────────────────────────────────────┐
│              第三步：智能调用                                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 匹配度判断：                                          │   │
│  │ • 高匹配（>70%）：直接调用技能                         │   │
│  │ • 中匹配（30-70%）：询问用户是否使用技能               │   │
│  │ • 低匹配（<30%）：继续常规处理，记录备选技能           │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

## 技能优先级规则

当多个技能可能适用时，按以下顺序处理：

### 优先级 1：流程控制类（决定"怎么做"）
| 技能 | 触发条件 |
|------|---------|
| brainstorming | 新功能、新项目的早期讨论 |
| debugging | 错误排查、问题诊断 |
| systematic-debugging | 复杂问题的系统性调试 |

### 优先级 2：执行指导类（指导"做什么"）
| 技能 | 触发条件 |
|------|---------|
| test-driven-development | 编写功能代码时 |
| code-review | 代码变更完成时 |
| documentation | 需要文档输出时 |

### 优先级 3：领域专用类（特定场景）
| 技能 | 触发条件 |
|------|---------|
| frontend-design | UI/前端开发 |
| mcp-builder | MCP 服务器开发 |
| api-design | API 接口设计 |

## 智能判断原则

### 应该调用技能的场景
- 用户明确请求某个功能/修复
- 任务涉及多个步骤或文件
- 需要遵循特定开发规范（如 TDD）
- 复杂度较高的技术决策

### 可以跳过技能的场景
- 简单的信息查询（"这个函数做什么？"）
- 问候或闲聊
- 用户明确表示"快速处理"或"跳过流程"
- 紧急修复（可事后补充流程）

## 用户控制

### 主动触发
用户可以通过以下方式主动触发技能：
- `/skill-name` - 直接调用特定技能
- "使用 [技能名] 来..." - 明确指定技能

### 主动跳过
用户可以通过以下方式跳过技能流程：
- "快速处理，跳过流程"
- "不要使用技能"
- "直接给我答案"

## 响应模板

### 调用技能时
```
🎯 检测到 [任务类型]，将使用 [技能名] 技能来处理。
   [技能简介]

如需跳过，请回复"跳过"或"直接处理"。
```

### 多技能匹配时
```
🔍 检测到多个相关技能：
   1. [技能A] - [简介]
   2. [技能B] - [简介]

推荐使用：[推荐技能]
请选择或回复编号，也可回复"跳过"直接处理。
```

## 与原技能的改进对比

| 方面 | using-superpowers | skill-orchestrator |
|------|-------------------|-------------------|
| 语气 | 强制性、命令式 | 引导性、协商式 |
| 灵活性 | 无条件执行 | 智能判断 + 用户可控 |
| 用户体验 | 可能过于繁琐 | 平衡规范与效率 |
| 平台支持 | Claude Code | Claude Code + OpenCode |
| 错误处理 | 无 | 有降级和跳过机制 |

## 配置选项

可在项目或用户级别配置以下选项：

```yaml
# .claude/skill-orchestrator.yaml
orchestrator:
  auto_invoke: true          # 自动调用技能
  confidence_threshold: 0.7  # 自动调用的置信度阈值
  allow_skip: true           # 允许用户跳过
  verbose: false             # 详细日志
```

## 注意事项

1. **不要过度调用**：技能是工具，不是目的。如果简单方法可行，不要强行使用复杂技能。

2. **尊重用户意图**：如果用户明确要求快速处理，优先响应用户需求。

3. **保持上下文**：在同一会话中，记住已使用的技能和当前流程状态。

4. **渐进式引导**：对于新手用户，可以主动建议使用技能；对于熟练用户，可以更低调。
